4. Результаты: `false`, `true`, `false`.
String Pool — системный кэш строк. Литерал "Hello" сохраняется в пуле напрямую, а оператор `new String()` форсированно резервирует новую подстроку памяти за его пределами. Вызов `.intern()` принудительно заставит JVM найти эквивалентную по содержанию строку в String Pool и вернуть конкретную ссылку на неё.

5. Код страдает "проблемой ромбовидного наследования" в интерфейсах. Два интерфейса содержат одинаковый default-метод, компилятор не может выбрать приоритет. Для исправления сам класс C должен жестко переопределить метод `hello()` внутри себя, явно разрешив конфликт.

6. `StringBuilder` — классический высокоскоростной мутатор, не имеющий синхронизации потоков. `StringBuffer` обладает полной внутренней потокобезопасностью за счет блокировок `synchronized`. `StringBuffer` оправдан только в ситуациях с расшаренной ссылкой на строковый буфер между разными параллельными потоками исполнения, которые собирают данные.

7. Выведет `10`. Если раскомментировать присвоение, компилятор выдаст ошибку. Локальные переменные, захваченные областью видимости лямбды, обязаны быть "эффективно константными" (effectively final), то есть не изменять первоначального значения. 

8. Развернутое прибавление: `s -> s + "текст"`. Ссылки на методы предполагают использование уже готового существующего метода типа-объекта в чистом виде (например, `String::trim`). Если в теле лямбды появляется дополнительное литеральное значение, оператор или создание нового объекта, заменить это на Method Reference невозможно.

9. `EnumMap` использует плоский массив, так как каждому элементу сопоставлено фиксированное целочисленное значение по методу `ordinal()`. Нет необходимости вычислять `hashCode` и обрабатывать коллизии хэш-таблиц.
